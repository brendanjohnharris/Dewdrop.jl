Benchmarks ​

Dewdrop is benchmarked against the established simulators (Brian2 and brian2cuda, BrainPy, NEST and NEST GPU, and GeNN) on the same network: a recurrent excitatory/inhibitory AdEx net with a fixed in-degree and current-based exponential synapses, the regime where the sparse synaptic scatter dominates. Every simulator builds an identical connectome (a language-agnostic splitmix64 sample, verified edge-for-edge) and is checked to produce statistically matching firing rates and CV-ISI before any timing is compared, so the numbers are apples-to-apples.

The panels show wall-clock time and peak memory as the network grows from 2,000 to 512,000 neurons, on the CPU (top) and GPU (bottom). Dewdrop's three CPU execution backends (Serial/Fused/Turbo) and its GPU Fused megakernel are each plotted separately; see choosing a backend.

What the scaling shows ​

  • CPU, at scale. Dewdrop's Fused backend is the fastest CPU simulator measured: at N=512k it is 3.4× faster than Brian2's compiled C++, 3.4× faster than BrainPy, and 15.6× faster than NEST, with no code-generation or compile step.

  • CPU, small networks. Fused (0.36 s) sits within a few percent of Brian2's hand-tuned C++ standalone (0.38 s) at N=16k: a pure-Julia engine at near-compiled-C++ throughput.

  • GPU, small networks. Dewdrop's GPU megakernel ties GeNN's generated CUDA (0.069 vs 0.067 s) and is roughly 2× faster than brian2cuda and 4-5× faster than NEST GPU: the launch-bound regime where the fused step removes per-phase kernel launches.

  • GPU, at scale. Dewdrop is fastest by a clear margin: at N=512k it is 2.2× faster than brian2cuda, 5.9× faster than NEST GPU, and 8× faster than GeNN, whose generated kernels degrade superlinearly there. It is also the leanest on the device at scale (1.3 GB).

  • Memory. On the CPU, Dewdrop is mid-pack (Brian2's standalone is the leanest, NEST by far the heaviest); on the GPU at scale, Dewdrop uses the least memory of the four.

The combination (competitive at small sizes on both devices and the fastest at scale) comes from the struct-of-arrays state, the fused single-pass step that pipelines on one device stream, and the compacted spike scatter that keeps the GPU off the memory-bandwidth wall (see running on the GPU).

Methodology ​

The shared problem (one spec.toml consumed by every simulator):

  • Network. Recurrent E/I AdEx, 80% excitatory, fixed in-degree K=100, CUBA exponential synapses with a constant external drive: the sparse-scatter-dominated regime.

  • Connectome. Generated by a language-agnostic splitmix64 algorithm so every simulator builds the identical graph; the comparison scripts verify the edge count matches across all simulators.

  • Run. dt = 0.1 ms, 500 ms for the throughput scan (one network size per octave from 2k to 512k), 1000 ms for the correctness run that checks firing rate and CV-ISI agree.

  • Measured. End-to-end wall time of the integration and peak resident memory, per simulator, per size, on CPU and (where supported) GPU. Absolute timings are machine-dependent; the scaling slopes and the relative ordering are the reproducible result.

Each simulator is implemented as its own self-contained runner under test/simulator_comparisons/<sim>/ (Dewdrop in Julia, Brian2/BrainPy/NEST/GeNN in Python), writing standard out/performance.csv files; compare_simulators.jl discovers them, verifies they ran the same simulation, and produces the figure above. Adding a simulator is dropping in a directory.

Reproducing ​

From the package's test/simulator_comparisons/ directory, run each simulator's runner and then the aggregator (see that directory's README.md and run_all.sh):

bash
julia --project=test dewdrop/run.jl        # Dewdrop, all sizes, CPU + GPU
python brian/run.py                        # each other simulator in its own environment
julia --project=test compare_simulators.jl # verify + plot to out/comparison.pdf
